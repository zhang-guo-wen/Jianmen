# 后端代码审计与重构记录

日期：2026-07-18

## 范围与方法

本次审计覆盖 `cmd/` 与 `internal/` 下 196 个 Go 文件，其中生产代码 133 个、测试文件 63 个。检查内容包括：

- 包职责和跨层依赖
- Handler 直接数据库访问
- Store 接口规模与 `context.Context` 使用
- 构造函数依赖表达和启动失败行为
- 单文件行数上限
- 授权、认证、资源分组的数据一致性
- 干净工作区下的编译和测试可用性

验证命令：

```text
go test ./... -count=1
go build ./...
go vet ./...
npm run typecheck
npm run build
```

## 审计结论

### P1：管理 API 仍有较多跨层数据库访问

审计开始时，`internal/server/admin` 有 21 个生产文件直接使用 GORM，共 167 处命中。首轮重构后降为 17 个文件、109 处命中。

剩余高密度文件：

| 文件 | 直接数据库命中 |
|---|---:|
| `temporary_access_handlers.go` | 26 |
| `user_group_handlers.go` | 17 |
| `user_handlers.go` | 14 |
| `init.go` | 13 |
| `resource_access.go` | 8 |
| `rbac.go` | 8 |

建议按“一个资源一条垂直切片”的顺序继续迁移到 `handler -> service -> store`，优先处理临时授权、用户组和用户管理。

### P1：Store 是过大的聚合接口

`store.Store` 当前有 78 个方法，其中 62 个方法没有把 `context.Context` 作为第一个参数。这会导致：

- 请求取消和超时无法向数据库传播
- Server、代理和测试依赖远超实际需要
- 修改一个资源接口时影响所有实现和 Mock

建议在使用方定义小接口，按认证、主机账号、数据库账号、审计、会话等资源拆分；迁移时同步把 `context.Context` 加到第一参数。

### P1：临时授权链路存在错误被忽略

`temporary_access_handlers.go` 中延长、禁用临时账号时，对授权、用户会话和 AI Token 的部分更新使用 `_ =` 忽略错误。多表状态可能出现不一致。

建议将这些操作移入 Service，并由 Store 在单个事务中完成；任何子更新失败都应回滚。

### P2：表现层和持久层类型仍有混用

- 多个 GORM Model 带 JSON 标签。
- `store.go` 同时承载数据库接口和 API View 类型。
- `internal/server/admin` 同时包含 Server、路由、Handler 和业务判断。

建议新增独立 DTO，逐步把 Handler 移到 `internal/handler/admin/`，Server 只保留启动、路由和中间件。

### P2：授权检查需要批量化

资源授权检查器仍直接依赖 GORM，且列表过滤需要逐个资源判断。当前重构已按资源键缓存单次请求内的判断结果，但数据量增大后仍应提供批量授权查询，避免 N+1 查询。

### P2：协议文件复杂度偏高

`internal/server/dbproxy/observer.go` 和 `mysql_auth.go` 接近或超过 500 行。它们尚未违反当前针对“代理会话文件”的 600 行硬限制，但职责已较密集，建议按协议解析、认证状态机、审计观察器拆分。

## 本轮已完成的重构

### 管理员认证

- 新增 `AdminAuthService` 和上下文感知的 Store 方法。
- Middleware 不再直接查询或更新 GORM。
- 区分无效 Token、账号过期和数据库故障。
- 账号过期禁用失败不再被静默忽略。

### 资源授权

- 新增 `ResourceGrantService` 和资源授权 Repository。
- Handler 不再执行 SQL。
- 非超级管理员只能列出自己有权管理的资源授权，避免全量授权关系暴露。
- 创建请求使用明确 DTO，不能注入 ID、创建时间等持久化字段。
- 补齐应用、容器端点的搜索和资源存在性检查。
- 所有查询错误均向上返回，不再忽略 `Count`/`Pluck` 错误。

### 资源分组

- 新增 `ResourceGroupService` 和资源分组 Repository。
- 分组列表计数、创建、修改、删除均移出 Handler。
- 资源分组改名和删除会同步容器端点。
- 删除分组会清理指向该分组的资源授权，避免悬空授权。
- 支持清空分组描述，并对重复名称返回冲突。
- 资源分组授权现在覆盖容器端点。

### 启动与构建

- Admin、SSH Server 构造函数改为显式必需依赖并返回错误。
- 移除可选的 `...*gorm.DB` 和初始化失败时的 `panic`。
- 将元数据初始化从 `main.go` 拆出，入口文件降至 109 行。
- 增加嵌入前端目录占位文件，干净检出后无需先生成前端产物即可编译和运行 Go 测试。
- 当前按项目规则检查的单文件行数违规为 0。

## 后续建议顺序

1. 临时授权：事务化延长、禁用、会话和 Token 状态更新。
2. 用户组：拆出成员管理 Service/Repository。
3. 用户管理与初始化：统一账号状态、有效期、超级管理员规则。
4. RBAC CRUD：移出 Admin Server，并把 Checker 改为接口依赖。
5. 按资源拆分 `store.Store`，逐步为全部方法加入 Context。
6. 将 API DTO 从 Model 和 Store 中迁出。
