# Jianmen 工作项台账

最后更新：2026-07-19

本文件用于统一管理需求、缺陷、安全问题、技术债和测试/运维改进，是当前工作项状态的唯一入口。历史审计和实施计划只作为证据，不直接代表当前状态；历史记录与本文件冲突时，以本文件最近一次复核结果为准。

## 1. 管理原则

1. 新发现的问题先登记、分级和确认范围，未经确认不得顺手实现。
2. P0/P1 可中断当前计划；P2 按当前切片的完成标准处理；P3 默认进入 backlog。
3. 每个切片最多一轮实现、一轮独立复审和一轮修复。仍有争议时拆成新工作项，不循环扩大范围。
4. 一个工作项对应一个可独立验收的小提交；禁止长期维持大规模未提交工作区。
5. 开发期间只运行目标包测试。全仓测试、Docker 实库测试和完整打包只在阶段提交及最终合并前执行。
6. 合并回 `dev` 前必须先合并最新 `dev`，解决冲突并完成适用于本次变更的门禁。
7. 完成、延期和取消都必须记录原因及证据，不能只删除工作项。

## 2. 类型与编号

| 前缀 | 类型 | 使用场景 |
|---|---|---|
| `REQ` | 需求 | 新的用户价值、业务能力或产品行为 |
| `DEF` | 缺陷 | 已有承诺行为错误、退化或不可用 |
| `SEC` | 安全问题 | 权限、认证、敏感数据、协议边界等安全风险 |
| `TECH` | 技术债 | 架构、可维护性、性能或迁移治理问题 |
| `TEST` | 测试改进 | 自动化覆盖、测试夹具或质量门禁缺口 |
| `OPS` | 工程运维 | 构建、发布、启动、监控和运维体验问题 |

编号格式为 `<类型>-<YYYYMMDD>-<三位序号>`。编号创建后不得复用。

## 3. 优先级

| 优先级 | 定义 | 处理要求 |
|---|---|---|
| P0 | 可直接导致 RCE、大范围权限绕过、不可逆数据损坏、核心系统完全不可用 | 立即停止其他工作，修复并独立复审 |
| P1 | 敏感数据泄露、有限范围权限绕过/数据损坏、核心链路无可接受替代方案 | 当前阶段阻断项，优先修复 |
| P2 | 重要功能或工程可靠性问题，有临时替代方案且无直接高危安全影响 | 当前切片内则清零，否则排入明确阶段 |
| P3 | 非关键增强、代码整洁度、穷举测试、体验或性能优化 | 默认延期，不阻断阶段交付 |

优先级按影响而不是实现难度确定。无法复现或证据不足的历史问题使用“待复核”状态，不直接按旧优先级阻断开发。

## 4. 状态流转

正常流程：

```text
待分诊 -> 待复核 -> 已确认 -> 待开发 -> 开发中 -> 复审中 -> 待验证 -> 已完成
```

旁路状态：

- `已阻塞`：外部依赖或必要决策未满足，必须写明阻塞条件。
- `已延期`：问题有效，但不进入当前阶段，必须写明重新评审时间或触发条件。
- `已取消`：不再符合产品方向、已被替代或收益不足，必须保留决策依据。

同一工作项只能有一个当前状态。开始开发前必须从“已确认”进入“待开发”，禁止直接从“待分诊”跳到“开发中”。

## 5. 工作项最小字段

每个工作项至少记录：

| 字段 | 要求 |
|---|---|
| ID / 标题 | 唯一编号和面向结果的简短标题 |
| 类型 / 优先级 / 状态 | 使用本文定义的枚举 |
| 所属阶段 | 当前阶段、下阶段或 backlog |
| 负责人 | 未分配时写“待分配” |
| 来源 | 用户反馈、审计、测试、日志或关联文档 |
| 影响范围 | 用户、资源类型、协议、数据或部署方式 |
| 当前行为 | 已观察到的事实，不写猜测 |
| 目标行为 | 修复或交付后应达到的结果 |
| 验收标准 | 可执行、可判定，至少包含正常路径和失败路径 |
| 验证证据 | 测试命令、结果、提交号或截图/日志位置 |
| 决策记录 | 延期、取消、拆分或优先级变化的原因 |

### 需求补充字段

- 用户价值和目标用户。
- 范围内与范围外内容。
- API、权限、审计、数据模型和连接路径影响。
- 交互或配置变化。

### 缺陷补充字段

- 复现前提和最小复现步骤。
- 实际结果与预期结果。
- 首次发现版本/提交。
- 回归风险和需要补充的自动化测试。

## 6. Ready 与 Done

### Definition of Ready

工作项进入“待开发”前必须满足：

- 影响和优先级已确认；
- 范围、非目标和验收标准明确；
- 依赖、数据迁移及安全影响已识别；
- 能拆成一个小提交，或已经拆成多个独立工作项；
- 明确目标测试，不要求开发过程反复运行全仓门禁。

### Definition of Done

工作项进入“已完成”前必须满足：

- 验收标准全部通过；
- 目标测试通过，错误路径有覆盖；
- 完成一轮独立复审及至多一轮修复；
- 没有把新发现的非 P0/P1 问题混入提交；
- 提交已记录，文档状态和验证证据已更新；
- 阶段合并时已合并最新 `dev` 并执行仓库要求的完整门禁。
